# DOCS FLOW - Reinos Medievais

**Descrição:**  
Use este workflow quando o objetivo for criar ou atualizar documentação em Markdown sobre mecânicas de jogo, regras econômicas, guias de desenvolvimento ou manuais de UI.  
A função deste workflow é transformar informações soltas em documentos estruturados e prontos para consulta.

---

## 🎯 Missão

Organizar o conhecimento do projeto em documentos Markdown úteis, claros e acessíveis para outros desenvolvedores ou para o próprio usuário.

---

## 🧭 Princípios

- **Clareza acima de tudo.**
- **Não inventar dados:** Documentar apenas o que existe ou o que foi definido.
- **Estrutura lógica:** Use títulos, listas e blocos de código.
- **Saída limpa:** O arquivo deve estar pronto para salvar.

---

## 📥 Quando Acionar

Acione este workflow quando quiser:

- criar um README.md ou atualizar um guia.
- documentar uma nova regra de combate ou economia.
- criar um manual de componentes UI.
- organizar a lista de tecnologias usadas.
- documentar fluxos de trabalho ou convenções.

---

## 📥 Entradas Esperadas

- tema do documento (ex: "Guia de Combate")
- pontos chave a cobrir
- público alvo (desenvolvedores, jogadores, etc.)
- nível de detalhe desejado

---

## 📤 Saídas Obrigatórias

- Documento Markdown estruturado.
- Sugestão de nome de arquivo.
- Resumo do conteúdo gerado.

---

## 🛠️ Protocolo de Execução

### Passo 1: Interpretar o Pedido

Identificar o tipo de documento:

- Guia Técnico
- Documento de Regras de Negócio
- Manual de Uso
- Referência de API/Estado

---

### Passo 2: Montar a Estrutura

Organizar em seções:

1. Título e Descrição
2. Objetivo
3. Conceitos Principais
4. Detalhes / Regras
5. Exemplos (se houver)
6. Considerações Finais

---

### Passo 3: Gerar o Conteúdo

Escrever de forma objetiva, usando formatação rica do Markdown (alerts, tabelas, mermaid diagrams se necessário).

---

## 🚫 Fronteiras

Este workflow:

- **não escreve código**
- **não altera a lógica do jogo**
- **não implementa funcionalidades**
- **não faz debug**

---

## ✅ Checklist Final

- [ ] O documento tem uma estrutura lógica?
- [ ] O conteúdo reflete a realidade do projeto?
- [ ] A linguagem está adequada ao público?
- [ ] O arquivo está pronto para ser salvo como .md?

---

## 📌 Formato de Saída Recomendado

````markdown
# [Título do Documento]

## Visão Geral
[Breve descrição]

## Regras de Funcionamento
- [Regra 1]
- [Regra 2]

## Exemplos de Uso
```javascript
// Exemplo de código ou dado
```

## Próximos Passos

[O que fazer com essa info]
````
